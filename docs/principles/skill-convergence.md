# 双 Agent 日报系统封装为 Skill 前的收敛原则

## 文档目的

这份文档用于记录一个重要判断：

当前这套双 agent AI 日报系统（Hermes + OpenClaw），**不应直接原样封装成 skill** 给外部使用，而应先做一轮“单 agent 化”的架构收敛，再进入 skill 封装。

这份文档的作用是：

- 把这个判断固定下来
- 防止后面直接跳进 skill 实施
- 作为未来做 skill 产品化前的前置原则

## 当前判断

### 结论

**先不要直接做 skill。**

正确顺序应该是：

1. 先做“单 agent 版日报系统”的架构收敛设计
2. 再基于收敛后的结构做 skill 封装

### 原因

当前系统虽然已经能跑，但它是围绕本地环境逐步演化出来的，具备大量上下文依赖：

- 双 agent 分工
- Hermes 与 OpenClaw 各自不同职责
- 多套 cron
- 双状态流转
- 观察面同步
- 快报与正式日报两条不同输出链
- 多次演化过程中加入的修补逻辑

如果直接原样封装成 skill，风险很大：

- 很可能做成“当前系统的打包版”
- 而不是“别人能直接复用的通用能力”

## 为什么不能直接原样封装

### 1. 当前系统的本地特例太多

包括但不限于：

- 本地路径结构
- 飞书应用配置
- 观察面表结构
- 通知目标
- agent 分工方式
- 历史演化中的状态字段和补丁逻辑

这些内容如果不先剥离，就会导致 skill 无法直接迁移。

### 2. 双 agent 对外部用户来说太重

如果使用者：

- token 不够
- 环境不完整
- 没有这套上下文

那么双 agent 版系统会过于复杂，外部 agent 很难稳定复刻。

### 3. 当前系统的正确方向已经变成“脚本主导 + 模型轻参与”

而这更适合做成：

- 单 agent
- 强脚本驱动
- 模型只做判断层

不是两个 agent 长期分治协作。

## 未来 skill 的目标形态

后续如果要封装 skill，目标形态应为：

### 单 agent

不再要求：

- Hermes 一个 agent
- OpenClaw 一个 agent

而是收敛成：

- 一个 agent
- 两种模式

### 模式 A：快报模式

- 一天多次
- 输出“最新值得关注的信息”

### 模式 B：正式日报模式

- 一天一次
- 对少量入选内容读原文
- 输出飞书 docx 正式日报

### 统一主库

- SQLite 仍然是唯一 source of truth

### 统一观察面

- 飞书多维表只作为观察面

## 当前正式原则

在 skill 封装前，必须先做下面这件事：

### 产出一份“单 agent 版日报系统收敛设计”

这份设计至少要回答：

1. 哪些能力必须保留
2. 哪些双 agent 分工要合并
3. 哪些本地特例必须剥掉
4. 哪些功能是可选开关
5. skill 最终交付的是：
   - 快报模式
   - 正式日报模式
   - 还是两者兼容

## 为什么现在先不做

当前优先级仍然是：

- Hermes 主链路稳定
- OpenClaw 正式日报稳定
- 事实层脚本化边界收口

也就是说：

**现在不是 skill 产品化时机，而是系统先稳定和收敛边界的时机。**

## 后续正确步骤

未来继续推进 skill 时，按这个顺序：

1. 先做单 agent 架构收敛设计
2. 评估快报/正式日报模式如何统一
3. 明确最小可交付能力
4. 再做：
   - SKILL.md
   - 配置模板
   - 脚本 scaffold
   - 安装与验收文档

## 一句话总结

当前双 agent AI 日报系统，**不应直接原样封装成 skill**。

正确做法是：

**先收敛成单 agent、脚本主导、模型轻参与的通用架构，再做 skill 产品化。**
