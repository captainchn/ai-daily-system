# AI 日报系统整体搭建方案（Hermes + OpenClaw）

## 文档目的

这份文档面向一个**没有上下文记忆的外部 agent**，目标是让它只靠这份文档，就能理解并复刻一套完整的 AI 日报系统。

系统目标：

1. 持续抓取 AI 资讯
2. 将资讯写入结构化主库
3. 同步一份飞书多维表作为观察面
4. 用 Hermes 在固定时间点发送“最新值得关注的信息”
5. 用 OpenClaw 每天生成一份正式飞书日报文档

## 一、系统分工

这套系统分成两条链路：

### 1. Hermes：精选提醒层

职责：

- 定时抓取 AI 资讯
- 写入 SQLite 主库
- 同步飞书多维表观察面
- 在 `08:00 / 12:00 / 16:00` 发送“最新值得关注的信息”

定位：

- 快报 / 提醒
- 不负责长文正式日报

### 2. OpenClaw：正式日报层

职责：

- 从 Hermes 主库读取结构化候选
- 做正式日报选题
- 对少量入选条目读原文
- 生成飞书 docx 正式日报
- 可附带封面图和消息通知

定位：

- 正式日报 / 长文产出

## 二、核心架构原则

### 原则 1：SQLite 是唯一 source of truth

不要把飞书多维表当主库。

主链路必须是：

- 抓取 → SQLite
- Hermes 快报 → 读 SQLite
- OpenClaw 正式日报 → 读 SQLite
- 飞书多维表 → 观察面

### 原则 2：事实层脚本化，判断层交给大模型

脚本负责：

- 标题
- 链接
- 来源
- 发布时间
- 抓取时间
- 候选池构建
- 去重
- 状态写回
- 飞书观察面同步

大模型负责：

- 选哪些条目值得关注
- 写摘要
- 写分析

不允许大模型自由改写：

- 标题
- 链接
- 来源
- 时间

### 原则 3：展示层和数据层分离

数据层保留真实字段，不为“展示好看”去改写。

展示层可以：

- 增加摘要
- 增加分析
- 增加封面
- 增加栏目组织

但展示层生成内容不得回写覆盖事实层字段。

## 三、技术栈

推荐技术栈如下：

### 主库

- SQLite

### 抓取与任务编排

- Python
- cron（Linux / WSL）

### LLM

- MiniMax `MiniMax-M2.7-highspeed`
- 建议显式 provider 调用，不走不透明黑箱兼容链路

### 观察面

- 飞书多维表（Bitable）

### 正式日报输出

- 飞书 docx / wiki

### 图片/封面（可选）

- Python 图像脚本
- 或其他稳定本地生成方式

## 四、数据流

完整数据流如下：

```text
信源抓取
  ↓
Hermes 脚本清洗 / 去重 / 分类 / 打分
  ↓
SQLite 主库（唯一事实源）
  ├─→ 飞书多维表观察面（可视化同步）
  ├─→ Hermes 快报（08:00 / 12:00 / 16:00）
  └─→ OpenClaw 正式日报（08:30）
            ↓
       飞书 docx 正式日报
            ↓
         群/私聊通知（可选）
```

### 标签字段

- `openclaw_status`
- `openclaw_selected_at`

### 审计字段

- `updated_at`

- `sync_status`
- `error_note`

## 六、时间字段语义

必须明确区分：

- `published_at`：原文发布时间
- `fetched_at`：系统抓到该条内容的时间

推荐使用方式：

- Hermes 快报窗口：按 `fetched_at`
- OpenClaw 正式日报窗口：按 `fetched_at`

原因：

- 系统关注的是“最近进入素材池、值得关注的内容”
- 而不是严格的“原网站昨天发布时间”

## 七、观察面（飞书多维表）

### 角色

飞书多维表只作为：

- SQLite 主库的可视化同步视图

### 推荐做法

- 新建一张“干净观察面表”
- 不在历史脏表上继续修补
- 旧表保留，只读，不再写入

### 观察面字段建议

- 标题
- 链接
- 平台
- 发布时间
- 原文发布时间
- 分类
- AI相关度
- 来源层级
- quality_score
- content_type
- 原文摘要 / 提取式摘要
- 去重指纹

## 八、Hermes 工作流

### 1. 抓取

标准节奏：

- `07:30 / 11:30 / 15:30`

抓取后：

- 写入 SQLite
- 可选同步观察面

### 2. 快报

标准节奏：

- `08:00 / 12:00 / 16:00`

Hermes 不负责长文，而是：

- 从最新候选中选出值得关注的信息
- 发送精选快报

### 3. 快报实现原则

Hermes 发送时采用：

- 脚本选事实
- LLM 做选择与摘要/分析

禁止：

- 让 LLM 改写标题
- 让 LLM自由生成条目

## 九、OpenClaw 工作流

### 1. 正式日报生成时间

- 每天 `08:30`

### 2. 正式日报链路

OpenClaw 的正式日报应当采用两层结构：

#### 第一层：标题层选题

OpenClaw 基于这些字段做初筛：

- 标题
- 分类
- 质量分
- 来源层级
- 时间
- 去重结果

先决定：

- 哪些内容值得进入正式日报候选集

#### 第二层：原文层写作

只对最终入选的少量条目：

- 打开原文
- 读取正文
- 写摘要
- 写分析

这样可以：

- 降低成本
- 降低幻觉
- 保持事实可信度

## 十、推荐工作流

### 1. Hermes 抓取与入库

- 抓取 → 清洗 → 分类 → 打分 → SQLite

### 2. Hermes 快报

- 基于 SQLite 选最新值得关注内容
- 输出快报

### 3. OpenClaw 正式日报

- 先根据标题做选题
- 再读原文写正式日报
- 输出飞书 docx

### 4. 观察面

- 与主库同步
- 提供人工查看和复盘能力

## 十一、验收标准

一套完整可用的系统，应至少满足：

1. SQLite 是唯一事实主库
2. 飞书多维表只是观察面
3. Hermes 能按 cron 正常抓取和发送
4. OpenClaw 能按 `08:30` 生成正式日报
5. 标题/链接/来源/时间 100% 来自数据库或原文
6. 发送前可校验，防止幻觉条目被发出
7. 观察面与主库字段口径一致

## 十二、一句话总结

这套 AI 日报系统的本质，不是“让大模型自动写日报”，而是：

**用脚本锁住事实层，用大模型做选择和表达，再用 Hermes 和 OpenClaw 分别承担提醒层与正式日报层。**
