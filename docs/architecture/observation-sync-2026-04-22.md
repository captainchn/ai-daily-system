# 观察面同步时机调整结论

## 文档目的

这份文档用于记录 2026-04-22 对“飞书观察面应在什么时候同步”的最终结论，作为后续 Hermes 观察面逻辑的正式语义基线。

这件事不是简单的代码改动，而是一个**产品语义选择**：

- 观察面究竟是“抓到什么就显示什么”
- 还是“这轮实际发出去什么，就显示什么”

## 最终结论

从 2026-04-22 起，Hermes 观察面的正式定义调整为：

### 观察面 = Hermes 已发送批次的可视化结果视图

不再定义为：

### 观察面 = 抓取后立刻同步的全量最新素材视图

也就是说：

- SQLite 仍然是全量事实主库
- 飞书多维表观察面是“已处理/已发送结果”的展示层

## 为什么要改

之前的实现是：

- `fetch_and_write.py` 在抓取写入 SQLite 后，顺手就同步多维表

这种方式的问题是：

1. **观察面会比 Hermes 实际发送状态更快**
   - 抓完就进表
   - 但这轮快报可能还没发，甚至发失败

2. **容易让观察面语义变得混乱**
   - 用户看到“今天已经有记录”
   - 但实际上提醒层还没完成

3. **主链路和观察面没有彻底解耦**
   - fetch 阶段还在承担飞书同步职责

因此，这种“抓完立刻同步”的模式不再适合当前系统定义。

## 当前正式设计

### SQLite 角色

SQLite 是：

- 唯一 source of truth
- 全量素材主库

只要抓取成功，数据就应该进入 SQLite。

### 观察面角色

飞书多维表观察面是：

- SQLite 的后置同步结果
- 用于查看“已处理/已发送”的可视化视图

### Hermes 主链路顺序

新的顺序是：

1. 抓取写入 SQLite
2. 生成 Hermes 快报
3. 实际发送成功
4. 发送成功后，再同步观察面

即：

- `fetch` 不再同步观察面
- `generate_short_report.py` 在 `send_ok=True` 后触发 `sync_to_bitable.py`

## 当前实现方式

### 1. fetch_and_write.py

已调整为：

- 只负责：
  - 抓取
  - 清洗
  - 写 SQLite
- 不再负责：
  - 飞书观察面同步

### 2. generate_short_report.py

已调整为：

- 发送成功后，调用 `sync_to_bitable.py`

即：

- 观察面同步被后置到发送完成之后

### 3. sync_to_bitable.py

继续保留为：

- 幂等
- 可补跑
- 不影响主链路

## 这次调整的直接效果

### 优点

#### 1. 主链路优先级更清楚

先保证：

- SQLite 写入
- Hermes 快报发送

再做：

- 观察面同步

#### 2. 观察面语义更清楚

从今以后，观察面不再表示“刚抓到的内容”，而表示：

- 这轮 Hermes 实际处理并发出的内容

#### 3. 排障更容易

以后如果有人说：

> 为什么 SQLite 里有，但观察面没有？

答案很清楚：

- 因为这轮还没成功发送，观察面本来就不会同步

## 需要明确接受的代价

这个调整不是没有代价的。

### 代价 1：观察面会比 SQLite 滞后

抓取成功后：

- SQLite 会立刻有数据
- 观察面不会立刻更新

必须等到 Hermes 快报实际发送成功后，观察面才更新。

### 代价 2：如果发送失败，观察面不会反映那批抓取结果

这不是 bug，而是当前选择的正式语义。

如果后续想要另一种语义，例如：

- 观察面同时承担“最新抓取池”作用

那就意味着需要重新定义产品目标。

在当前版本中，不这么做。

## 观察面统计口径的补充结论

这次排查中还发现：

- 不应继续依赖飞书 filter API 的 `today / gte` 结果判断是否同步成功

原因是：

- `lark-cli` 在某些 filter 传参情况下会出现“code=0 但 items=[]”的假空结果
- 容易误判为“今天没有同步进去”

因此，从现在开始，观察面统计口径统一为：

### 统一口径

1. `GET /records` 拉全量记录
2. 用 Python 本地按时间字段过滤

不再把飞书 filter API 作为同步是否成功的主要判断依据。

## 当前正式语义

从现在开始，观察面在系统中的正式定义是：

### SQLite

- 全量事实主库

### 飞书观察面

- Hermes 已发送批次的可视化结果视图

两者不是一回事，也不要求完全同一时刻同步可见。

## 一句话总结

2026-04-22 起，Hermes 观察面同步正式后置到发送完成之后。

这意味着：

- SQLite 负责“抓到了什么”
- 观察面负责“这轮实际发出了什么”

这样虽然牺牲了一部分实时性，但换来了更清晰的语义、更稳的主链路，以及更一致的用户理解。
